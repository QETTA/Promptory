# Promptory 사업·제품 기준 문서

상태: 내부 기준 문서  
최종 수정일: 2026-04-01  
목적: 프롬프토리가 외주형 커스텀 개발이 아니라 반복 가능한 제품 회사로 가도록 기준을 고정한다

## 0. 문서 사용 규칙

### 상태 표기 규칙

- `[확정]` : 지금 팀이 기준으로 잠가야 하는 판단
- `[가설]` : 실제 판매와 운영으로 검증해야 하는 상업 가설
- `[정렬 필요]` : 현재 레포 화면, 카피, 가격, 패키지 구성이 이 문서와 아직 맞지 않는 항목

### 우선순위 규칙

현재 단계에서는 이 문서의 기준이 개별 랜딩 카피나 일시적인 화면 문구보다 우선한다.  
화면이 다른 말을 하고 있다면 화면을 이 문서에 맞춰 고친다.

## 1. 사업 정체성

### 한 문장 정의

프롬프토리는 Slack을 쓰는 중소규모 B2B 팀이 URL이나 업무 자료를 넣으면 진단, 짧은 질문, 실행 초안까지 이어서 받아볼 수 있는 맞춤형 Slack Agent Package다.

### 내부에서 잠가야 할 문장

우리는 AI 자체를 파는 회사가 아니다.  
우리는 팀 대화 안에서 초안을 더 빨리 만들게 해주는 회사다.

### 프롬프토리가 실제로 파는 것

- 마케팅팀과 사업기획팀의 초안 작성 병목 해결
- Slack 안에서 끝나는 진단 -> 질문 -> 초안 흐름
- 공통 엔진 위에 회사별 설정을 얹는 반제품화 구조

### 프롬프토리가 아닌 것

- 범용 AI 플랫폼
- 마켓플레이스 중심 제품
- 광범위한 자동화 툴
- 회사마다 새로 만드는 Slack 봇 개발 대행
- 완전 무인형 범용 에이전트

## 2. 진짜 고객

### 1차 ICP

Slack을 일상적으로 쓰는 20인에서 200인 사이의 B2B 중소기업에서, 흩어진 자료를 대표 보고용 초안이나 실행용 문서로 바꿔야 하는 마케팅 리드 또는 사업기획 리드.

### 가장 좋은 첫 구매자

- 마케팅 리드
- 사업기획 리드
- 대표 질문을 대신 정리해야 하는 팀장
- 보고용 요약과 비교표를 자주 만드는 실무 책임자

### 이 ICP가 중요한 이유

- 구매자와 사용자가 거의 겹친다
- 고통이 반복적이고 분명하다
- Slack이 이미 업무 표면으로 깔려 있다
- 몇 달짜리 구축 없이도 빠르게 가치 검증이 가능하다

### 구매가 발생하는 순간

- "사이트가 왜 전환이 안 되냐"는 질문을 받았을 때
- 회의는 끝났는데 브리프가 아직 없을 때
- 내일 미팅인데 경쟁사 비교표가 급할 때
- 대표나 HQ에 보낼 5줄 요약이 필요할 때

### 맞지 않는 고객

- Slack을 거의 쓰지 않는 조직
- 초안 가속이 아니라 완전 자동화를 원하는 조직
- 깊은 시스템 연동 없이는 가치를 못 느끼는 조직
- 마케팅이나 기획 결과물의 오너가 없는 조직

## 3. 고객이 돈을 내는 문제 정의

### 핵심 문제

고객의 진짜 문제는 정보 부족이 아니다.  
고객의 진짜 문제는 `자료는 있는데 초안이 없다`는 것이다.

### 고객이 실제로 반복해서 미루는 일

- 사이트 병목 진단
- 회의 후 캠페인 브리프 정리
- 경쟁사 비교표 작성
- CTA와 카피 초안 만들기
- 대표 보고용 요약 정리

### 우리가 약속하는 변화

프롬프토리는 흩어진 자료를 팀 대화 안에서 바로 쓸 수 있는 첫 초안으로 바꾼다.

### 현재 기준에서 잠근 판단 `[확정]`

- 첫 판매 메시지는 `AI`보다 `초안 병목 해결`이 우선이다
- 첫 전면 패키지는 `Website Diagnosis Agent`와 `Campaign Brief Agent` 두 개다
- `Competitor Scan`은 독립 제품보다 후속 출력 또는 업셀 워크플로가 더 적합하다
- `Korea Local Ops Agent`는 P1 또는 프리미엄 패키지로 다루는 편이 안전하다
- 파일 입력은 허용하되, 처음부터 모든 파일을 완벽히 읽는 제품처럼 약속하지 않는다

## 4. 제품 구조 원칙

### 올바른 제품 모델

프롬프토리는 이렇게 팔아야 한다.

`업무별 Agent Package + 회사별 설정`

프롬프토리는 이렇게 팔면 안 된다.

`회사별 맞춤 Slack 빌드`

### 왜 회사별 빌드는 위험한가

- 고객마다 요구가 달라져 매번 새로 만들어야 한다
- 매출이 늘수록 사람을 더 붙여야 한다
- 코드 재사용이 줄어든다
- QA가 흔들린다
- 제품이 아니라 프로젝트 사업이 된다

### 왜 업무별 패키지는 맞는가

- 같은 엔진을 여러 회사에 반복 적용할 수 있다
- 세팅 중심으로 납품할 수 있다
- UX와 QA를 공통화할 수 있다
- 영업 메시지와 온보딩 구조를 표준화할 수 있다

### 반제품화 구조 정의

고정되는 층:

- 입력 구조
- Slack 인터랙션 패턴
- 질문 수 제한
- 출력물 종류
- 저장과 이어보기 방식
- 사람 검수 규칙

회사별로 바뀌는 층:

- 목표 선택지 문구
- 질문 플로우 세부 문장
- 출력 강조 순서
- 용어와 톤
- 공유 채널 방식
- 보고 포맷 선호

고객은 "우리 회사에 맞춰졌네"라고 느껴야 한다.  
내부 팀은 "같은 패키지에 설정만 다르다"라고 이해해야 한다.

## 5. 제품 경험 구조

### 기본 흐름

1. 사용자가 URL, 붙여넣은 메모, 파일 자료를 보낸다
2. 에이전트가 현재 작업 종류를 식별한다
3. 에이전트가 최대 2~3개의 짧은 질문으로 우선순위를 고정한다
4. 에이전트가 하나의 메인 결과물을 생성한다
5. 사용자는 버튼이나 후속 선택지로 관련 결과물을 더 요청한다
6. 작업은 저장되어 다음 주에도 이어서 쓸 수 있다

### 반드시 나와야 하는 결과물 타입

- 진단 요약
- 비교표
- 카피 초안
- 보고용 요약
- 액션 리스트

### 제품 차원의 규칙

- 한 번에 한 개의 메인 결과물을 중심으로 준다
- 질문은 짧고 의사결정 중심이어야 한다
- 결과물은 Slack, 이메일, 문서에 그대로 옮기기 쉬워야 한다
- 읽을 수 없는 링크나 자료는 읽을 수 없다고 말해야 한다
- 위험도가 높은 출력은 사람 검수 1회를 허용한다

### 입력 정책 `[확정]`

입력 우선순위는 아래 순서를 기본으로 한다.

1. 공개 URL
2. 붙여넣은 텍스트 메모
3. 텍스트 추출이 가능한 문서 파일
4. 이미지 위주 또는 구조가 약한 자료

운영 규칙:

- 텍스트가 충분한 입력일수록 자동 결과물 신뢰도를 높게 본다
- 이미지 위주 입력이나 로그인 막힌 링크는 fallback 입력으로 전환한다
- 파일 지원은 `모든 파일을 읽는다`가 아니라 `읽을 수 있는 자료를 우선 읽는다`로 설명한다

## 6. 패키지 구조

### 전체 판매 구조

프롬프토리는 아래 3단 구조로 판다.

1. Quick Audit
2. Core Slack Agent Package
3. Upsell 레이어

### 기본 계약 구조 `[확정]`

제품을 팔 때는 `패키지 단위`와 `세팅 단위`를 분리해서 설명한다.

패키지 단위는 고정한다:

- 어떤 업무를 해결하는지
- 어떤 입력을 받는지
- 질문이 몇 개인지
- 어떤 결과물이 나오는지
- 어떤 Slack 표면에서 동작하는지

세팅 단위는 고객별로 조정한다:

- 목표 선택지 문구
- 질문 세부 문장
- 결과물 템플릿 강조
- 내부 공유 방식
- 용어와 톤

### 범위 오염 방지 규칙 `[확정]`

아래 항목은 기본 패키지 범위 밖으로 본다.

- 별도 워크스페이스 추가
- 새 업무를 위한 별도 패키지 설계
- 깊은 CRM 또는 광고 계정 연동
- 완전 커스텀 리포트 시스템 구축
- 다국가/다부서 동시 운영 설계

이 항목들은 업셀 또는 별도 견적으로 다룬다.

### 6.1 Quick Audit

역할:

- 유료 진입 상품
- 신뢰 형성
- 급한 문제를 빠르게 정의
- Core Package 전환 전 검증

약속하는 결과:

- URL 또는 자료 1세트 기준 빠른 진단
- 핵심 병목 요약
- 1차 비교 또는 카피 방향
- 대표 보고용 짧은 정리

상업적 메모:

- 아래 가격은 아직 확정값이 아니라 가설이다
- 현재 작업 가설: 29만원에서 79만원 사이

기본 범위 `[확정]`:

- 회사 1곳 기준
- URL 또는 자료 1세트 기준
- 목적 1개 기준
- 결과물 1패킷 기준
- Slack 설치 없이 결과 제공 가능
- 추가 조사나 후속 템플릿은 별도 전환 대상으로 넘긴다

### 6.2 Core Slack Agent Package

역할:

- 메인 매출원
- 반복 사용이 가능한 주간 워크플로
- 고객 조직 안에 자리잡는 운영 제품

상업적 메모:

- 아래 가격도 아직 확정값이 아니라 가설이다
- 현재 작업 가설: 구축 300만원에서 500만원
- 현재 작업 가설: 월 운영 79만원에서 149만원

기본 범위 `[확정]`:

- Slack 워크스페이스 1개
- 기본 패키지 1종
- 질문 플로 1세트
- 핵심 출력 포맷 1세트
- 최근 작업 저장과 이어보기
- 운영 초기 세팅과 사용 가이드

### 6.3 Upsell 레이어

확장 가능한 항목:

- KR/EN 동시 요약
- 경쟁사 스캔 확대
- 대표 보고용 템플릿
- 추가 워크플로 패키지
- 검수와 운영 지원 강화

## 7. 우선 패키지 라인업

### P0 패키지

#### Website Diagnosis Agent

적합한 상황:

- 대표가 사이트 전환 문제를 묻는 상황
- 사이트 URL만으로 빠른 병목 진단이 필요한 상황
- Quick Audit으로 가장 빨리 팔 수 있는 상황

입력:

- 공개 사이트 URL
- 짧은 텍스트 메모
- 필요 시 보조 파일 자료

질문:

- 목표 선택: 문의, 리드, 구매
- 필요 시 대상 고객이나 전환 기준 보정

출력:

- 5줄 병목 진단
- CTA 초안
- 비교용 포인트
- 대표 보고용 한 문단

왜 중요한가:

- 즉시 가치가 보인다
- 판매 진입 장벽이 낮다
- Quick Audit 전환에 가장 적합하다

#### Campaign Brief Agent

적합한 상황:

- 회의는 끝났는데 브리프가 없는 상황
- 메모와 자료는 있는데 우선순위가 섞여 있는 상황
- 실행팀에 넘길 1차 문서가 필요한 상황

입력:

- 회의 메모
- 붙여넣은 Slack 스레드 텍스트
- 랜딩 URL
- 문서 파일 자료

질문:

- 목표 선택: 리드 확보, 인지도, 신청 전환
- 필요 시 타깃이나 일정 보정

출력:

- 1장 브리프
- 핵심 메시지 정리
- CTA 문구 초안
- 액션 리스트
- 공유용 5줄 요약

왜 중요한가:

- 반복성이 높다
- 월 운영비 모델과 잘 맞는다
- 사용 빈도가 쌓일수록 락인이 생긴다

### P1 워크플로와 프리미엄 레이어

초반 전면 상품으로 밀기보다, P0 검증 후 확대해야 하는 항목:

- Competitor Scan 워크플로
- Korea Local Ops Agent
- App Home 이어보기 강화

### 반드시 기억할 점

링크 막힘 대응과 우선순위 충돌 해결은 별도 상품이 아니다.  
모든 패키지에 기본으로 들어가야 하는 핵심 기능이다.

## 8. 수익 모델

### 구조

프롬프토리는 아래 3단계로 돈을 번다.

1. Quick Audit으로 문제를 유료로 정의한다
2. Core Package로 월 반복 매출을 만든다
3. Upsell로 계정당 매출과 마진을 키운다

### 이 구조가 맞는 이유

- 무료 상담보다 유료 진단이 고객의 긴급도와 적합도를 빨리 드러낸다
- 패키지 + 세팅 구조는 반제품화 모델과 잘 맞는다
- 초안 작업은 매주 반복되기 때문에 월 운영비 정당성이 있다
- 업셀은 새 제품 서사를 만들지 않고도 매출을 키울 수 있다

### 수익 모델 운영 원칙

- 커스텀 범위 산정 자체를 무료 컨설팅으로 주지 않는다
- 사람 검수가 아직 필요한 단계에서 순수 SaaS 가격처럼 싸게 팔지 않는다
- 큰 맞춤 구축을 먼저 팔지 않는다
- AI 성능이 아니라 반복 워크플로 가치로 가격을 설명한다

### 검증해야 할 수익 가설 `[가설]`

- 고객은 `무료 데모`보다 `유료 Quick Audit`에서 더 강하게 진지한 전환을 보인다
- 첫 고객은 패키지 1종만으로도 충분히 가치를 느낀다
- 첫 3개 유료 고객 중 2개 이상은 월 운영 또는 후속 작업으로 이어질 수 있다
- 가격 설명은 기능 수보다 `대표 보고 시간을 줄여준다`, `브리프 초안을 바로 만든다` 같은 결과 중심 표현이 더 잘 먹힌다

## 9. 하지 않을 것

### 제품 차원에서 하지 않을 것

- 회사별로 완전히 새 Slack 봇을 만들지 않는다
- P0에서 AI 플랫폼 서사를 전면에 내세우지 않는다
- P0에서 마켓플레이스 서사를 전면에 내세우지 않는다
- P0에서 카카오톡, 왓츠앱 등 멀티채널 확장을 벌리지 않는다
- P0에서 광고 계정이나 CRM 깊은 연동을 선행 조건으로 두지 않는다
- P0에서 브라우저 자동 조작형 실행 에이전트를 약속하지 않는다
- 사람 검수 없는 완전 무인 자동화를 약속하지 않는다
- 핵심 흐름에서 질문을 3개 이상 늘리지 않는다
- 읽지 못한 링크를 읽은 척하지 않는다
- 결과물을 여러 장 한꺼번에 덤프해서 사용성을 해치지 않는다

### 운영 차원에서 하지 않을 것

- 반복 가능한 패키지 정의 없이 새 상품을 만들지 않는다
- 고객 요구 하나 때문에 전체 아키텍처를 새로 짜지 않는다
- 엔터프라이즈급 연동이 없으면 가치가 안 나오는 구조로 만들지 않는다

## 10. P0 구현 범위

### P0 목표

첫 3개 유료 고객에게 실제로 팔고, 실제 Slack 흐름 안에서 써볼 수 있는 최소 제품을 만든다.

### P0에서 반드시 완성할 워크플로

1. Website Diagnosis
2. Campaign Brief
3. 링크가 막혔거나 표면 정보가 약할 때의 대체 입력 플로
4. 같은 스레드 안에서 우선순위가 충돌할 때의 정리 플로

### P0 입력 범위

- 공개 URL
- 짧은 텍스트 메모
- 붙여넣은 회의 요약
- 읽을 수 있는 문서 파일 자료

### P0 대체 입력 범위

- 핵심 내용 5줄 붙여넣기
- 파일 업로드
- 공개 URL 다시 보내기

### P0 질문 정책

- 최대 2~3개 질문
- 가장 먼저 목표를 고정한다
- 긴 인터뷰형 탐색은 메인 흐름에서 제외한다

### P0 결과물 범위

- 진단 요약
- 브리프 또는 액션 메모
- 비교용 정리
- CTA 초안
- 보고용 짧은 요약

### P0 저장 범위

최소 저장 상태:

- 원본 URL 또는 자료 참조
- 선택한 목표
- 패키지 종류
- 마지막 결과물 제목
- 마지막 결과물 시각

이 정도면 무거운 메모리 기능 없이도 "지난번 작업 이어서 하기"를 구현할 수 있다.

### P0 사람 검수 범위

아래 항목은 사람 검수 1회를 허용한다.

- 경쟁사 비교에서 민감한 주장
- KR/EN 동시 보고용 요약
- 대표, 고객, HQ 등 대외 공유 비중이 높은 문서

### P0 검증 지표 `[가설]`

초기 운영에서 아래 지표를 본다.

- 자료 입력 후 첫 의미 있는 결과물까지의 시간
- 질문 2~3개 플로 완료율
- 생성 결과물이 그대로 공유된 비율
- Quick Audit 이후 후속 문의 또는 패키지 상담 전환율
- 첫 사용 후 14일 안에 재사용이 발생하는 비율

초기 해석 원칙:

- 산출물 품질보다 먼저 `실제로 공유됐는지`를 본다
- 사용자가 추가 설명 없이 재사용하면 워크플로 적합도가 높다고 본다
- 질문 단계에서 이탈이 많으면 제품이 아니라 질문 설계가 문제일 가능성이 높다

### 현재 레포 기준 P0 표면

현재 저장소에서는 아래 표면을 P0의 중심으로 본다.

- `/` : 제품 포지셔닝과 전환 진입점
- `/optimize` : 실제 작업이 일어나는 핵심 표면
- `/demo/slack` : 영업용 시연 표면
- `/packages/*` : 패키지 설명과 적합도 판단 표면
- `/pricing`, `/contact` : 상업 전환 표면

## 11. 교차검수 결과와 현재 레포 정렬 필요 항목

아래 항목은 현재 문서와 실제 레포 사이의 가장 큰 충돌 지점이다.

### 11.1 패키지 우선순위 불일치 `[정렬 필요]`

현재 홈, 데모, 패키지 목록, 푸터, 공통 콘텐츠는 `Korea Local Ops Agent`를 `Website Diagnosis Agent`, `Campaign Brief Agent`와 같은 1차 전면 패키지처럼 보여주고 있다.

대표 충돌 위치:

- `src/app/page.tsx`
- `src/app/demo/slack/page.tsx`
- `src/app/packages/page.tsx`
- `src/components/layout/site-footer.tsx`
- `src/lib/promptory-content.ts`

기준 문서상 잠금:

- 1차 전면 패키지는 2개
- `Korea Local Ops Agent`는 P1 또는 프리미엄 포지션

정리 방향:

- 홈과 패키지 메인에서는 2개 패키지를 전면에 둔다
- `Korea Local Ops Agent`는 별도 산업/시장형 옵션이나 프리미엄 라인으로 내린다

### 11.2 Quick Audit 가격과 무료 데모 서사 충돌 `[정렬 필요]`

현재 레포는 `Quick Audit`을 무료 데모처럼 보이게 하거나, `4.9~9.9만` 가격대로 표기하고 있다. 이건 현재 사업 기준의 `유료 진입 상품` 가설과 충돌한다.

대표 충돌 위치:

- `src/app/pricing/page.tsx`
- `src/app/contact/page.tsx`
- `src/app/page.tsx`
- `src/app/proposal/page.tsx`
- `src/app/packages/website-diagnosis-agent/page.tsx`
- `src/app/packages/campaign-brief-agent/page.tsx`
- `src/app/packages/korea-local-ops-agent/page.tsx`
- `src/lib/promptory-content.ts`

기준 문서상 잠금:

- `Quick Audit`은 유료 진입 상품으로 검증한다
- 가격은 아직 확정이 아닌 가설이며, 현재 가설은 `29만~79만`

정리 방향:

- 무료 데모와 유료 진입 상품을 분리해서 설명한다
- `무료 Quick Audit` 문구는 제거하거나 `무료 상담/샘플 리뷰`로 낮춘다
- 가격은 하나의 공식 기준으로 통일한다

### 11.3 너무 이른 확장 기능 노출 `[정렬 필요]`

현재 가격 페이지는 다중 에이전트, API 연동, HubSpot, Airtable 같은 고급 옵션을 비교적 이른 단계부터 전면에 보여준다. 이는 `P0에서는 깊은 연동과 멀티플로를 선행 조건으로 두지 않는다`는 기준과 어긋난다.

대표 충돌 위치:

- `src/app/pricing/page.tsx`

기준 문서상 잠금:

- 첫 판매는 `초안 병목 해결`이어야 한다
- 깊은 연동은 후순위 업셀이다

정리 방향:

- 가격표 전면은 P0 패키지와 유료 진입 구조만 남긴다
- API 연동과 고급 옵션은 `추가 협의 항목` 정도로 낮춘다

### 11.4 제품 메시지의 무게중심 보정 필요 `[정렬 필요]`

현재 일부 화면은 아직 `맞춤형 Slack Agent Package`라는 범주 설명에 더 무게가 있다. 기준 문서는 `초안 병목 해결`과 `자료는 있는데 초안이 없다`를 더 강한 헤드라인 축으로 본다.

대표 보정 대상:

- `src/app/page.tsx`
- `src/app/packages/page.tsx`
- `src/app/pricing/page.tsx`
- `src/app/demo/slack/page.tsx`

정리 방향:

- `에이전트`보다 `초안`, `브리프`, `비교표`, `대표 보고용 요약`을 먼저 보여준다
- 카테고리 설명보다 실제 결과물 이름을 먼저 배치한다

### 11.5 아직 공식 약속으로 내보내면 안 되는 상업 문구 `[정렬 필요]`

현재 가격/제안 화면 일부에는 아직 검증되지 않은 운영 약속이 들어가 있다.

대표 충돌 위치:

- `src/app/pricing/page.tsx`
- `src/app/proposal/page.tsx`

대표 문구 예시:

- `무제한 URL 진단 및 초안 생성`
- `일반적으로 2~4주 소요`
- `팀 단위 요금제로 인원 수와 무관`
- `연간 선납 시 2개월 무료`

문제점:

- 아직 첫 3개 유료 고객 기준의 운영 데이터가 없다
- 무제한, 고정 기간, 연간 할인은 한 번 외부에 약속하면 되돌리기 어렵다
- 가격 정책보다 먼저 혜택 문구가 고정될 위험이 있다

정리 방향:

- 실제 운영 데이터를 얻기 전까지는 `별도 협의`, `초기 고객 기준 맞춤 제안` 정도로 낮춘다
- `무제한` 대신 `패키지 운영 범위 내 반복 사용`처럼 보수적으로 표현한다
- 할인 정책과 세팅 기간은 내부 검증 뒤 공식화한다

## 12. 실행 순서

### 당장 해야 할 순서

1. Website Diagnosis Agent 흐름을 잠근다
2. Campaign Brief Agent 흐름을 잠근다
3. Quick Audit 유료 진입 페이지와 제안 문구를 만든다
4. 링크 막힘 대응과 우선순위 충돌 해결 로직을 핵심 흐름에 넣는다
5. 패키지 수를 넓히기 전에 3개 회사에 실제로 팔아본다

### P0 완료 기준

P0는 화면이 예뻐졌다고 끝나는 것이 아니다.  
P0는 아래가 되면 끝이다.

- 실제 사용자가 자료를 넣을 수 있다
- 제품이 짧은 질문으로 목표를 고정한다
- 바로 쓸 수 있는 초안을 돌려준다
- 결과물을 대표나 팀에 다시 손대지 않고 공유할 수 있다
- 이전 작업을 이어서 불러올 수 있다
- 실제 유료 전환 대화에서 돈을 낼 의향이 확인된다

## 13. 의사결정 필터

새 기능이나 새 요청이 들어오면 아래 5개 질문으로 자른다.

1. 이 기능이 반복되는 초안 병목을 줄이는가
2. Slack 또는 가벼운 연결 표면에서 시작할 수 있는가
3. `입력 -> 질문 2~3개 -> 결과물 1개` 구조 안에 들어오는가
4. 회사별 재개발이 아니라 설정 변경으로 재사용 가능한가
5. 이 기능이 패키지 판매나 유지율에 직접 도움이 되는가

대부분의 답이 아니오면 뒤로 미룬다.

## 14. 마지막으로 기억할 문장

프롬프토리는 회사마다 새로 만드는 Slack 봇 사업이 아니다.

프롬프토리는 업무를 제품화한 뒤,  
그 회사가 자기 회사용처럼 느끼도록 설정해서 파는 사업이다.

### 팀 내부 고정 문장

우리는 회사를 처음부터 커스텀하지 않는다.  
우리는 업무를 제품화하고, 회사를 거기에 맞춘다.
